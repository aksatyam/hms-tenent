OPD — Outpatient Department

Patient entry point — UHID creation, EMR consultation, e-prescription generation, and seamless cross-module event dispatch via Kafka.

Pharmacy → Lab → Finance → Portal → Audit
🏥
Module Purpose

The OPD module is the primary entry point for all patient interactions. It handles walk-in registration, appointment check-in, digital queue management, clinical consultation with SOAP-based EMR, e-prescription with real-time drug interaction checking, lab order dispatch, and final discharge with automated billing. All events are published to Kafka for downstream module consumption.

1
Patient Registration & UHID Generation
Walk-in or appointment check-in triggers registration. System queries existing UHID by name + DOB + mobile — returning patients auto-populated. New patients: demographics (name, DOB, gender, address), insurance details, known allergies, and emergency contact captured. UHID generated as tenant-scoped unique identifier (format: TENANTID-YYYYMMDD-SEQ). Consent form presented and digitally signed (DISHA compliant).
PostgreSQL Write L3–L5 Required patient.registered
2
Token & Digital Queue Management
Digital token issued for the selected doctor/department. WebSocket pushes live queue updates to lobby display screens and reception desk. Queue system supports: Regular, Emergency, Senior Citizen (65+), and Pregnant Woman priority lanes. Patient receives SMS with token number and estimated wait time. Doctor's workstation shows real-time patient queue with current token highlighted.
Socket.IO Live SMS Alert token.issued
3
Doctor Consultation — SOAP EMR
Doctor opens patient record showing: past visit history, active medications, known allergies, lab history, and vitals trend. Consultation recorded in SOAP format (Subjective, Objective, Assessment, Plan). Vitals entry: BP, pulse, temperature, SpO2, weight, height, BMI auto-calculated. ICD-10 diagnosis picker with search and multi-diagnosis support. Allergy warnings cross-checked against drug master in real-time. Consultation notes auto-saved with 30-second draft intervals.
EMR Write Doctor Role Auto-save WS
4
E-Prescription & Lab Order Generation
Doctor selects drugs from formulary — drug interaction engine fires in real-time for any combination (Rxnorm cross-reference). Dosage, frequency, duration, and instructions captured per drug. Prescription finalised → Kafka event prescription.created published → Pharmacy worklist updated within <1 second via Socket.IO. Lab orders created with test names, specimen type, urgency flag, and clinical notes → laborder.created Kafka event → Lab worklist populated instantly. Referral orders to other specialists tracked separately.
prescription.created laborder.created <1s Propagation Drug Interaction API
5
Discharge & Automated Bill Generation
Doctor marks consultation complete → discharge summary auto-populated from EMR (diagnosis, prescribed drugs, investigations done, follow-up instructions). All charges aggregated: consultation fee + pharmacy charges (from drug.dispensed event) + lab charges (from result.validated event) → single consolidated bill in Finance module. Patient notified via SMS/WhatsApp with discharge summary PDF link (secure, time-limited presigned S3 URL). Next appointment suggestion prompted based on diagnosis.
discharge.generated SMS + WhatsApp S3 PDF Upload

Pharmacy & Drug Management

FIFO dispensing engine, real-time drug interaction alerts, automated stock reorder, and seamless billing integration via Kafka event consumption.

OPD (Prescriptions) Finance → Inventory → Audit
💊
Module Purpose

The Pharmacy module receives prescriptions in real-time via Kafka, manages drug dispensing with FIFO batch logic, enforces allergy and drug-interaction safety checks, maintains full batch traceability, automatically posts drug charges to Finance, and triggers procurement reorder workflows when stock falls below threshold. Multi-branch inter-transfer capabilities ensure stock availability across the hospital group.

1
Prescription Receipt & Worklist Update
Kafka consumer listens on prescription.created topic. On event receipt, prescription details decoded: patient UHID, drug list, dosage, duration, prescribing doctor. Pharmacist worklist updated via Socket.IO — prescription appears at top of queue within <1 second. Urgent prescriptions (flagged by doctor) highlighted in red with audio alert. Walk-in OTC requests handled separately through manual entry workflow. Prescription printed/displayed on pharmacist screen with patient allergy banner.
prescription.created (consume) Socket.IO <1s Pharmacist (L5)
2
Drug Availability Check & FIFO Batch Selection
System queries drug master for each prescribed item. FIFO engine auto-selects oldest expiry batch first — batch number, expiry date, supplier, and MRP displayed. If requested quantity unavailable in current branch → inter-branch transfer check performed across all branches in tenant. Drug-allergy cross-check against patient allergy profile (from UHID record) fires automatically — alerts shown with severity level. Drug-drug interaction matrix checked for all prescribed combinations. Partial fulfillment workflow: pharmacist can dispense available portion, pending items flagged for follow-up.
FIFO Query Interaction Check Live Stock
3
Dispensing Confirmation & Stock Ledger Update
Pharmacist reviews drug list, batch allocations, and patient details → confirms dispensing. Stock ledger decremented atomically in PostgreSQL transaction (prevents oversell race conditions). Full traceability record created: patient UHID + batch number + expiry + supplier + pharmacist ID + timestamp. Label printed with patient name, drug, dosage instructions in vernacular if configured. Controlled substance dispensing triggers mandatory justification entry and additional approval flag for Branch Admin.
Atomic Stock Debit drug.dispensed Controlled Drug: L4 Approval
4
Automated Billing to Finance Module
On dispensing confirmation, drug.dispensed Kafka event published with: patient UHID, visit ID, drug name, quantity, unit price, total amount. Finance module's Kafka consumer picks up event and adds drug charge line to patient's running bill — zero manual entry. For cashless insurance patients, TPA pre-authorisation reference attached to charge line. Receipt generated and displayed/printed for patient. Inpatient pharmacy charges tracked separately per ward/day for IPD billing.
drug.dispensed → Finance Zero Manual Billing Receipt Print
5
Procurement Reorder & PO Workflow
After every dispensing, system compares current stock against reorder level set per drug per branch. If stock ≤ reorder level → auto purchase requisition created in Inventory module via API call. Store Manager notified by in-system alert and email. Branch Admin approves requisition → PO auto-generated and emailed to preferred vendor from rate contract. GRN workflow in Inventory updates pharmacy stock when goods received. Expiry alerts run nightly: drugs expiring within 90/60/30 days highlighted in dashboard for return-to-vendor action.
Inventory API Email PO to Vendor Reorder Trigger

Laboratory & Diagnostics

Barcode-tracked sample management, HL7/ASTM analyzer integration, dual-validation pathology workflow, and instant report delivery to EMR and patient portal.

OPD (Orders) OPD EMR → Finance → Patient Portal →
🔬
Module Purpose

The Laboratory module handles the complete diagnostics lifecycle: test order receipt from OPD or walk-in, sample collection with barcode tracking, HL7/ASTM analyzer interface for auto-capture of results, multi-tier pathologist validation with critical value escalation, branded PDF report generation, and secure delivery to the OPD EMR and patient portal simultaneously.

1
Test Order Receipt & Worklist Population
Lab order from OPD arrives via Kafka laborder.created event (patient UHID, ordered tests, urgency, clinical notes, prescribing doctor). Walk-in requests entered manually with UHID lookup. Lab technician worklist populated automatically — sorted by urgency and order time. Test TAT commitments loaded from test catalogue (CBC: 2h, culture: 72h, etc.). External referral orders from other labs tracked with promised TAT. Patient informed via SMS/portal of expected report time.
laborder.created (consume) Live Worklist TAT SMS
2
Sample Collection & Barcode Tracking
Lab technician calls patient for sample collection. Unique barcode label printed per specimen (blood, urine, culture, etc.) with patient name, UHID, test name, and timestamp. Sample tube labelled and scanned — status changes to COLLECTED. Multi-specimen tests managed: one order can generate multiple labelled tubes. Sample tracked through status stages: Collected → In Transit → Received at Lab → In Analysis → Completed. Lost sample workflow: re-collection request raised with justification. Temperature-sensitive samples flagged with handling instructions.
Barcode Scan Log Status Tracking
3
Analysis & Result Entry (HL7/ASTM)
HL7/ASTM interface connects to lab analyzers (Sysmex, Mindray, Beckman Coulter, etc.) — results auto-captured via HL7 v2.x or ASTM E1394 protocol and mapped to order line items. For analyzers without interface, manual result entry with unit and reference range pre-filled from test catalogue. System compares result against age/gender-specific reference ranges — values outside range automatically flagged as L (Low), H (High), LL (Critical Low), HH (Critical High) with visual colour coding. Multiple result types supported: numeric, text, ratio, dilution, morphology descriptors.
HL7/ASTM Protocol Reference Range
4
Pathologist Validation & Critical Value Escalation
Results submitted for validation queue. Dual-validation workflow: junior technician reviews → senior pathologist approves for complex panels. Critical values (HH/LL flags) require mandatory senior pathologist digital sign-off before release. On critical result validation → result.critical.alert Kafka event → duty doctor receives mandatory in-app notification requiring acknowledgement within 15 minutes. Non-acknowledged critical alerts escalate to Dept Head. Amended reports tracked with previous/current values and reason for amendment. All validations timestamped with validator identity for audit trail.
result.critical.alert Doctor Notification Senior Pathologist Sign-off
5
Report Delivery — EMR, Portal & Finance
After validation, branded PDF report auto-generated: hospital logo, patient demographics, test panel, results with reference ranges, interpretation notes, pathologist signature and credentials, QR code for result verification. Kafka event result.validated published → OPD EMR updated via WebSocket (lab result tab lights up on doctor's screen) + Patient portal push notification + secure time-limited S3 download link via SMS. Finance module billed automatically from labcharge.generated event. Report delivered to referring doctor email if external referral. Elasticsearch indexed for BI analytics.
result.validated → OPD + Portal WebSocket EMR Update SMS Download Link S3 PDF Storage

Finance & Billing Engine

Auto-aggregating multi-module bill engine with GST compliance, TPA/insurance pre-authorisation, multi-mode payment gateway, and day-end close automation.

All Modules (Charges) BI Reports → Patient Portal →
💰
Module Purpose

Finance is the central billing aggregator. All charges from OPD, Pharmacy, Lab, and IPD flow in automatically via Kafka events — zero manual charge entry. The module handles insurance TPA pre-authorization, multi-mode payment collection (cash, card, UPI, bank transfer), GST invoice generation, GSTR-1 export, refunds, credit notes, advance management, and automated day-end close with branch consolidation reports.

1
Automated Charge Aggregation from All Modules
Finance Kafka consumers subscribe to charge events across all modules: consultation.fee (OPD), drug.dispensed (Pharmacy), labcharge.generated (Lab), bed.rent.daily + procedure.done (IPD). Each event adds a line item to the patient's running bill automatically. Charge master validates: service code, category, GST slab, applicable discounts. Advance payments (pre-admission deposits for IPD) tracked separately and adjusted at billing. Tariff plans support multi-tier pricing: General, Insurance, CGHS, corporate empanelment rates — auto-applied based on patient category.
Multi-event Consumption Zero Manual Entry Charge Master
2
Insurance / TPA Pre-Authorisation
For insurance patients, TPA master database queried: insurer name, TPA name, policy number validation, coverage limits, excluded items. Pre-authorisation request raised electronically through TPA portal API integration (or manual letter for non-API TPAs). Pre-auth reference number recorded against bill. Cashless pathway: approved amount reserved, patient pays only co-pay/excess. Reimbursement pathway: full bill raised, claim documents generated (discharge summary, investigation reports, itemised bill). Pre-auth expiry tracked — renewal reminder raised 24h before expiry for long IPD stays.
TPA Portal API Pre-auth Tracking Renewal Alert
3
Multi-Mode Payment Collection
Cashier screen displays itemised bill with subtotal, discount applied, GST breakdown, advance adjustment. Payment modes: Cash (change calculation), Card/POS (integrated terminal), UPI QR (dynamic QR per bill, Razorpay/Paytm), Net Banking, NEFT/RTGS. Split payment across modes supported (e.g., ₹5000 cash + ₹3000 UPI). Patient advance deducted from bill. Payment confirmation triggers payment.received Kafka event → Audit log + BI analytics. Receipt generated: itemised + GST-compliant + unique receipt number.
Payment Gateway payment.received Receipt Print
4
GST Invoice Generation & Compliance
On payment, GST invoice auto-generated with: hospital GSTIN, patient details, itemised service list, HSN/SAC codes, CGST + SGST breakdown per slab (0%/5%/12%/18%), total tax amount, place of supply. Tax invoice compliant with Indian GST rules — sequential invoice numbering with branch prefix. GSTR-1 export: monthly export of B2B and B2C invoices in GSTN-portal JSON format. Invoice PDF uploaded to S3, pre-signed download link emailed. Cancelled bills: credit note issued with reason — GSTR-1 adjustment handled. Exempt services (blood transfusion, dialysis) flagged correctly.
GST Computation GSTN Export Email Invoice
5
Day-End Close & Branch Consolidation
At day-end, cashier initiates day-close sequence: cash drawer count entered and matched against system total — variance report if mismatch. Cashier session locked — no further transactions allowed for the day. Branch-level summary generated: OPD collections, IPD collections, pharmacy, lab, outstanding (credit/TPA). day-close.done Kafka event triggers BI nightly aggregation. Tenant Admin receives consolidated cross-branch summary email. Discrepancy workflow: cashier supervisor must sign off before session locks. Petty cash management: advances, reimbursements tracked separately. Month-end reconciliation report for accountant.
day-close.done Tenant Admin Email Branch Admin Approval

Biometric attendance integration, leave lifecycle management, automated payroll computation with PF/ESIC/PT deductions, and full Indian regulatory compliance reporting.

Finance → BI Reports → Audit
1
Employee Onboarding & Master Setup
HR executive creates employee record: personal details, Aadhaar + PAN (encrypted storage, DISHA compliant), academic qualifications, medical council registration number (for clinical staff), emergency contact. Designation, grade, and pay band assigned. Branch and department mapped. System credentials provisioned automatically based on role template — minimising access provisioning time. Biometric enrollment: fingerprint/face scan data registered in ZKTeco/Suprema device and employee ID mapped. Bank account for salary credited. PF and ESIC number generated or existing number recorded.
Employee Master Auto Role Assignment Biometric API
2
Shift Scheduling & Biometric Attendance
Branch Admin configures shift templates: morning, evening, night, rotational — with grace period, overtime threshold, and weekly off rules. Duty roster published 7 days in advance — staff notified via in-app and WhatsApp. Biometric API (ZKTeco/Suprema REST API) syncs punch records every 5 minutes: IN/OUT timestamps mapped to employee. Late, early-out, and absent flags auto-raised with threshold rules. Manual override with justification field for missed punches — requires Dept Head approval. Real-time attendance dashboard for Dept Heads. OT computation auto-triggered when hours exceed threshold.
Biometric Sync 5min attendance.marked Dept Head Override
3
Leave Management Lifecycle
Employee applies for leave: selects type (CL/SL/EL/Comp-off/Maternity/Paternity), dates, reason, and handover notes. Approval workflow: Dept Head approves first → Branch Admin has override authority → cross-branch resource check for critical roles. Leave balance auto-deducted from accrual ledger on approval. Sandwich leave rule applied: working days between holidays counted as leave. Leave encashment at year-end or resignation: calculated and sent to payroll run. Medical leave: doctor certificate upload required after 2 consecutive days. Team calendar shows leave visibility for Dept Heads. Maternity/Paternity: statutory period auto-calculated, legal compliance enforced.
Leave Ledger Multi-level Approval In-App + WhatsApp
4
Monthly Payroll Run & Computation
Payroll officer initiates monthly run. Engine aggregates: working days from attendance, approved leaves, OT hours, advances taken. Gross computation: Basic + HRA + TA + Special Allowances + OT pay. Deductions: PF (12% employee + 12% employer), ESIC (0.75% employee + 3.25% employer), Professional Tax (state-specific slab), TDS (Form 16 computation), salary advance recovery. Net salary calculated. Payslip PDF generated: all components, deductions, net pay, YTD figures. Batch salary disbursement file generated in NEFT/bank format. payroll.run Kafka event → Finance module for accounting entries.
payroll.run → Finance Statutory Computation Payslip Email
5
Statutory Compliance Reports
Monthly: PF challan (ECR format for EPFO portal upload), ESIC contribution statement (Form 5 for ESIC portal). Quarterly: TDS returns (Form 24Q) with PAN validation. Annually: Form 16 generation per employee (Part A from TRACES, Part B auto-computed), Form 16A for contract staff. Labour law compliance: registers maintained digitally — Form A/B (attendance register), Wage Register, Muster Roll. Professional Tax remittance challan per state. Full audit trail: all payroll changes with before/after values, approver, timestamp.
EPFO/ESIC Portal Statutory Reports Compliance Reminders

Inventory & Procurement

Multi-level PO approval, vendor management with rate contracts, GRN quality control, inter-branch transfers, and fixed asset lifecycle management.

Pharmacy (Stock) Finance → Audit
1
Reorder Alert & Purchase Requisition
Real-time stock monitoring: every dispensing or consumption event decrements the item ledger. When stock ≤ reorder level (configurable per item per branch), auto purchase requisition (PR) created. Store Manager and Dept Head notified via in-system alert + email. PR contains: item code, current stock, reorder qty, preferred vendor, estimated value. Emergency PRs (stock = 0) escalated immediately to Branch Admin with SMS alert. Multiple items can be bundled into a single PR. Last 3 purchase rates displayed for cost benchmarking.
Stock Monitoring Store Manager Alert stock.low (consume)
2
Multi-Level Approval Workflow
Approval matrix configurable by tenant: PR up to ₹10k → Store Manager, ₹10k–₹50k → Branch Admin, ₹50k+ → Tenant Admin. Each approver sees full PR details, current stock, consumption trend, and vendor quotes. Approver can: Approve, Reject with reason, or Modify quantity. Multi-item PR approval: all items approved or each line item independently. Approval audit trail: approver name, timestamp, decision, comments. Escalation: auto-escalation after 24h of inactivity at any approval stage. Emergency procurement: Branch Admin can bypass approval for critical items with mandatory post-approval within 24h.
Multi-Level Approval Email + In-App Approval Matrix
3
Vendor Management & PO Generation
Vendor master: registered vendors with GST number, PAN, bank details, category (pharma/surgical/equipment), performance rating, rate contracts. Rate contract items: PO price auto-populated from contract. New item purchase: 3-quote comparison mandatory above threshold — quotes uploaded as documents. PO auto-generated on final approval: sequential PO number, delivery date, payment terms, GST details, terms & conditions. PO dispatched to vendor via email API (Sendgrid/SES) + WhatsApp for preferred vendors. PO amendments tracked with version history. PO cancellation workflow with vendor notification.
PO Email to Vendor Rate Contract Vendor Portal API
4
GRN — Goods Receipt & Quality Check
Goods received at store: GRN created against PO number. Item-by-item verification: quantity received vs ordered, batch numbers, expiry dates (rejection if expiry < 6 months for pharma), physical condition. Quality check by pharmacist for drugs: random sample inspection per defined SOP. Accepted items: stock ledger incremented → Pharmacy real-time stock updated → Finance notified for payables. Rejected items: return-to-vendor document generated, vendor debit note created. Partial GRN supported: pending items tracked for follow-up. Vendor invoice reconciled against PO and GRN — three-way match for payment processing.
Stock Increment grn.completed → Finance Pharmacy Stock Sync
5
Asset Register & Consumption Tracking
Capital equipment tracked in fixed asset register: serial number, purchase date, cost, department, warranty, vendor. Depreciation schedule auto-computed (SLM/WDV method). AMC (Annual Maintenance Contract) alerts: reminder 30/60 days before expiry. Breakdown record linked to asset: downtime tracked, vendor escalation auto-raised if SLA breached. Inter-branch transfer of assets with custody transfer record. Consumable items tracked by department-wise consumption: OT gloves, IV sets, disposables. Monthly consumption reports by department for budgeting. Slow-moving/dead stock alerts for rationalisation.
Asset Register AMC Reminder Dept Consumption

Business Intelligence & Reports

Real-time Kafka event ingestion, nightly ETL aggregation, executive dashboards, scheduled report delivery, and Power BI / Tableau export APIs.

All Modules (Events) Elasticsearch → Dashboards →
1
Real-Time Event Stream Ingestion
All module Kafka events consumed by Analytics microservice → normalised event schema → written to Elasticsearch event index. Events carry: tenantId, branchId, moduleId, eventType, userId, entityId, timestamp. Used for live operational dashboards (queue lengths, active IPD occupancy, today's collections). Redis cache serves real-time dashboard widgets with <200ms latency.
2
Nightly ETL Aggregation
Triggered by day-close.done Kafka event. Spring Batch job aggregates events into summary tables per branch and tenant: daily revenue, OPD volume, bed occupancy, pharmacy turnover, lab TAT compliance, staff attendance. Pre-computed aggregations stored in summary tables. Redis cache refreshed from summary tables → dashboard load time <100ms. Historical data partitioned by tenant-month for query isolation.
3
Executive Dashboard — Real-Time View
Tenant Admin dashboard: cross-branch revenue comparison, OPD patient volumes, bed occupancy heatmap, doctor performance metrics, pharmacy turnover, outstanding bills, KPI trend lines. Branch Admin view: daily collections, department-wise OPD, staff attendance, consumable consumption rate. Department Head view: patient queue, consultation time averages, pending lab results, staff on-duty. All charts interactive — drill-down from tenant → branch → department → individual.
4
Scheduled Report Delivery
Report scheduler: daily, weekly, monthly, quarterly. Tenant Admin configures: report type, recipients, format (PDF/Excel/CSV), delivery time. Cron job generates reports → stored in S3 with pre-signed URL → email delivered via SES. Reports include: P&L summary, MIS reports, compliance reports (DISHA audit trail), drug consumption, lab TAT analysis, staff productivity. Failed delivery retried 3 times with alerting.
5
BI Tool Export & Webhook Streaming
REST API for external BI tools: Power BI connector, Tableau data source, Looker Studio integration. API returns structured JSON with RBAC-filtered data — tenantId scope enforced at API layer. Excel/CSV export with column selection. Webhook streaming: tenant can register webhook URL to receive real-time event stream (for custom analytics pipelines). Rate-limited to prevent abuse. API key management with expiry and rotation workflows.

Patient Portal & PWA

DISHA-aligned self-service portal — OTP authentication, appointment booking, report access, online payments, and multi-channel push/SMS/WhatsApp notifications.

1
Self-Registration & DISHA Consent
Patient visits portal URL or installs PWA. Mobile OTP verification (Twilio/MSG91) → profile creation: name, DOB, gender, address, blood group, known allergies. Linked to existing UHID if already a hospital patient (UHID lookup by mobile + DOB). DISHA consent form presented with granular toggles: data sharing, research use, guardian consent for minors. Consent version and timestamp stored immutably. Profile photo optional. ABHA (Ayushman Bharat Health Account) linking supported.

Failed login attempts: 5 consecutive → account lockout + Admin alert. Suspicious patterns: login from new device/IP → OTP re-verification required. Privilege escalation monitor: temporary elevated access with mandatory justification and time-bound expiry. Patient consent lifecycle: consent given → stored with version + timestamp. Right-to-access: patient requests full data export → system generates FHIR bundle within 7 days. Data retention policies: active patient data retained indefinitely, discharged patient data anonymised after 7 years per policy. Daily per-tenant PostgreSQL schema backup: pg_dump → AES-256 encrypted → S3 cross-region replication (Mumbai + Singapore). Retention: 30 daily, 12 weekly, 7 yearly backups. Monthly DR drill: restore to staging environment and validate data integrity. Redis snapshot backup: every 15 minutes to S3. Elasticsearch snapshot: daily to S3. Recovery runbook documented and version-controlled.
5
Vulnerability Management & Penetration Testing
OWASP ZAP: automated weekly scans against staging environment — OWASP Top 10 checks. Snyk CVE monitoring: continuous scanning of container images and dependencies — critical CVEs auto-alert DevOps. Annual 3rd-party penetration test by CERT-In empanelled agency. Bug bounty program for responsible disclosure. 72-hour breach disclosure workflow: detection → assessment → regulatory notification (CERT-In + DISHA authority) → public communication if required. Patch management SLA: Critical: 24h, High: 7d, Medium: 30d. Security training: mandatory quarterly awareness for all staff.
